# HOD Multi-Tenant Security Audit Plan

## Current Status
After reviewing the codebase, I found that most queries properly include college_id filtering for tenant isolation. However, several findById() operations in related resource lookups are missing college_id filtering, creating potential cross-tenancy vulnerabilities.

## Queries Reviewed
- Teacher controller: All queries properly include college_id ✓
- Department controller: All queries properly include college_id ✓
- Timetable controller: Most queries include college_id, but some findById operations are missing it
- TimetableSlot controller: Most queries include college_id, but some findById operations are missing it
- TimetableException controller: Exception queries include college_id, but related timetable lookups don't

## Security Findings
Multiple instances of findById() operations without college_id filtering:
1. Timetable.publishTimetable() - line 103: `Timetable.findById(req.params.id)`
2. Timetable.deleteTimetable() - line 698: `Timetable.findById(id)`
3. TimetableSlot.updateSlot() - lines 164, 170: 
   - `TimetableSlot.findById(slotId)`
   - `Timetable.findById(slot.timetable_id)`
4. TimetableSlot.deleteTimetableSlot() - lines 244, 250:
   - `TimetableSlot.findById(slotId)`
   - `Timetable.findById(slot.timetable_id)`
5. TimetableException.updateException() - line 516: `Timetable.findById(exception.timetable_id)`
6. TimetableException.deleteException() - line 516: `Timetable.findById(exception.timetable_id)`
7. TimetableException.approveException() - line 516: `Timetable.findById(exception.timetable_id)`
8. TimetableException.rejectException() - line 516: `Timetable.findById(exception.timetable_id)`

## Remaining Vulnerabilities
The main vulnerability is that while the initial resource lookup includes college_id filtering, subsequent lookups of related resources (like fetching a timetable by its ID from a slot or exception) do not include college_id filtering. This creates a potential bypass where:
1. An attacker who knows or can guess a valid ID from another college
2. And has that ID reference a resource that exists in their own college (to pass initial checks)
3. Could potentially access unauthorized data

However, successful exploitation would require:
- Knowledge of valid resource IDs from other colleges
- Those IDs to correspond to existing resources in the attacker's college
- Or additional vulnerabilities to bypass initial checks

## Cross-Tenant Access Possible?
YES - Theoretically possible through ID guessing or if IDs are leaked through other mechanisms, but requires specific conditions to be met in practice.

## Fixes Needed
Add college_id filtering to all related resource lookups:
1. In timetable.controller.js:
   - publishTimetable: Add college_id to findById
   - deleteTimetable: Add college_id to findById
2. In timetableSlot.controller.js:
   - updateSlot: Add college_id to both slot and timetable lookups
   - deleteTimetableSlot: Add college_id to both slot and timetable lookups
3. In timetableException.controller.js:
   - All exception operations: Add college_id to timetable lookups
