# Password Reset Workflow Audit Plan

## Current Password Reset Flow
1. User requests password reset via email
2. System validates email exists (User/Student/Teacher)
3. System checks rate limit (max 3 OTPs per hour per email)
4. System checks for existing active OTP (returns if found)
5. System generates 6-digit OTP, hashes it with bcrypt, stores in PasswordReset collection with 10-minute expiry
6. System attempts to send OTP via email (using college-specific SMTP config), continues even if email fails
7. User provides OTP and new password
8. System verifies OTP (compares plaintext with bcrypt hash)
9. On success: updates user password, blacklists all access tokens, revokes refresh tokens, marks OTP as used
10. Expired OTPs are cleaned up via TTL index (expireAfterSeconds: 0) and daily cleanup job

## Security Findings
- OTP is hashed with bcrypt before storage (not plaintext)
- Rate limiting prevents brute force (3 attempts/hour/email)
- OTP expiry is automatic via TTL index and application logic
- Used OTP cannot be reused (isUsed flag)
- After password reset, all tokens are invalidated (force re-login)
- OTP is never returned in API responses (security by design)

## OTP Storage Method
- OTP is hashed using bcrypt (salt rounds: 10) before storage in `otpHash` field
- Plaintext OTP is never persisted; only hash is stored
- Comparison uses bcrypt.compare() method

## SMTP Status
- SMTP configuration is college-specific, stored in CollegeEmailConfig model
- System uses `getCollegeTransporter()` to create Nodemailer transporter
- If college SMTP not configured, throws error but OTP generation still succeeds (email sending is non-blocking)
- Actual SMTP working status depends on individual college configuration

## Remaining Gaps
1. No account lockout after excessive failed OTP verifications (only rate limiting on generation)
2. Password reset does not require user to be logged in (by design, but could be enhanced with additional verification)
3. No logging of failed OTP verifications for security monitoring
4. No password history enforcement on reset
5. No requirement for current password when changing password via reset flow (by design)

PARTIAL
