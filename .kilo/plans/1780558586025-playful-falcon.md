# Workflow Audit: College Suspend & Restore

## 1. Suspend Logic

`college.controller.js` `deleteCollege` (line 106) calls `College.findOneAndUpdate` with `{ isActive: false }`. This triggers the `pre("findOneAndUpdate")` hook in `college.model.js` (line 69).

**Cascade behavior on suspend:**
- All models linked by `college_id`: `isActive → false` (Departments, Courses, Teachers, Subjects, FeeStructures, StudentFees, Notifications, Timetables, TimetableSlots, AttendanceSessions, AttendanceRecords, Users)
- DocumentConfigs by `collegeCode`: `isActive → false`
- **Students:** Only `status: "APPROVED"` → `status: "INACTIVE"` (lines 99–105). PENDING / REJECTED / ALUMNI are intentionally untouched.
- `college.isActive` itself is set to `false`.

## 2. Restore Logic

`college.controller.js` `restoreCollege` (line 154) calls `College.findOneAndUpdate` with `{ isActive: true }`. This triggers the second `pre("findOneAndUpdate")` hook in `college.model.js` (line 175).

**Cascade behavior on restore:**
- All models linked by `college_id`: `isActive → true` (Departments, Courses, Teachers, Subjects, FeeStructures, StudentFees, Notifications, Timetables, TimetableSlots, AttendanceSessions, AttendanceRecords, Users)
- DocumentConfigs by `collegeCode`: `isActive → true`
- **Students:** In `college.model.js` lines 202–208, the restore filter is `{ college_id, status: "INACTIVE" }` → `{ status: "APPROVED" }`. This sets **every** INACTIVE student to APPROVED.
- `college.isActive` itself is set to `true`.

**Student status lifecycle implied by the hooks:**

| Pre-suspend status | After suspend | After restore (current) |
|---|---|---|
| APPROVED | INACTIVE | APPROVED (correctly restored) |
| PENDING | PENDING (unchanged) | PENDING (unchanged) ✓ |
| REJECTED | REJECTED (unchanged) | REJECTED (unchanged) ✓ |
| ALUMNI | ALUMNI (unchanged) | ALUMNI (unchanged) ✓ |
| DEACTIVATED | INACTIVE | **APPROVED (BUG)** |

## 3. APPROVED students after restore

APPROVED students are set to INACTIVE by the suspend hook, then restored to APPROVED by the restore hook. **Correct.**

## 4. PENDING students after restore

PENDING students are untouched by both hooks. They remain PENDING. **Correct.**

## 5. REJECTED students after restore

REJECTED students are untouched by both hooks. They remain REJECTED. **Correct.**

## 6. DEACTIVATED students after restore

If a student was already `DEACTIVATED` before suspend, the suspend hook sets them to INACTIVE (no effective change). The restore hook then sees `status: "INACTIVE"` and sets them to `APPROVED`. **This is incorrect.** Those students should remain DEACTIVATED (or INACTIVE), not become APPROVED.

## 7. Are statuses preserved correctly?

**PARTIALLY BROKEN.** The two hooks are not symmetrical:
- Suspend: only touches `status === "APPROVED"` → `INACTIVE`.
- Restore: touches `status === "INACTIVE"` → `APPROVED`, with no way to distinguish INACTIVE students who were previously APPROVED vs. students who were originally DEACTIVATED or any other status that happened to be INACTIVE.

This means DEACTIVATED students are incorrectly promoted to APPROVED. It also means any future status added as INACTIVE would be incorrectly overridden.

## 8. Is User.isActive synchronized correctly?

On **suspend**: ALL users are set `isActive: false` (line 161-163).  
On **restore**: ALL users are set `isActive: true` (line 264-266).  
This matches the college lifecycle at a coarse level, but it has two issues:
- It reactivates users of any role (including STUDENT, TEACHER, HOD, etc.) regardless of whether their own entity was independently deactivated.
- In `college.middleware.js` (line 70-73), non-SUPER_ADMIN users are blocked if `college.isActive` is false, so `User.isActive` already blocks login/token usage independently for most roles. Reactivating all Users on restore is broadly safe but coarse-grained.

---

## Unauthorized Access Risk

`college.middleware.js` (line 63-73) is the main access guard. It checks `college.isActive` **at request time**. If a college is suspended:
- Non-SUPER_ADMIN users get a `403 COLLEGE_SUSPENDED` response on `collegeId` lookup middleware.
- The middleware already re-verifies by looking up the College document, not relying solely on `User.isActive`.
- The SUPER_ADMIN role bypasses the check (line 9-11), which is intentional but means a SUPER_ADMIN can still query/data-fetch while college is deactivated. No hard exposure of other tenants was observed. **Low risk.**

---

## Final Verdict

### Overall: PARTIAL

The suspend/restore cascade is **asymmetrical at the Student level**, causing:

1. **BUG (P1):** DEACTIVATED students are incorrectly promoted to APPROVED on restore because the restore hook blindly maps `status: "INACTIVE"` → `APPROVED`. This happens for any student whose status happened to be INACTIVE when the college was suspended, not just those converted from APPROVED → INACTIVE by the suspend hook.
2. **RISK (P2):** `User.isActive` is bulk-updated on both suspend and restore for all roles, which is coarse-grained. While not directly exploitable, it could prematurely re-enable accounts that were purposefully deactivated for other reasons.
3. **Caller experience risk:** The `findOneAndUpdate` hooks run silently. The controller sends a success response before the hook errors are observable in the response. If a parallel update fails mid-Promise.all, the error is thrown but the outer controller may not handle it cleanly (though `next(error)` is present).

### Priorities

- **P1:** Fix the Students restore filter to use a more specific identity (e.g., a separate `suspendedByCollege` flag or track suspended student IDs) so only APPROVED→INACTIVE students are restored to APPROVED. The current `status === "INACTIVE"` selector is too broad.
- **P2:** Consider whether `User.isActive` bulk updates should be narrowed (e.g., only users whose role entity exists, or exclude users independently suspended).
