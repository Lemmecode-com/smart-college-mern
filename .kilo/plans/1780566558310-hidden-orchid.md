# P0 Investigation: Student Admission & Enrollment Workflow

## Current Student Lifecycle

1. **Registration**: 
   - Student submits registration form via `/register/:collegeCode` endpoint
   - System creates User account first (with hashed password)
   - System creates Student document with status: "PENDING"
   - Student receives registration success email

2. **Approval Process**:
   - College admin/admission officer reviews pending students
   - Upon approval:
     - Ensures student has user_id (creates if missing)
     - Validates course and admission capacity
     - Creates StudentFee record
     - **Generates enrollment number** (format: `{collegeCode}-{courseCode}{admissionYear}-{sequence}`)
     - Updates student status to "APPROVED"
     - Creates parent/guardian accounts (if emails provided)
     - Sends admission approval email with enrollment number and login URL

3. **Rejection Process**:
   - College admin/admission officer rejects pending student
   - Updates student status to "REJECTED"
   - Records rejection reason
   - Sets reapplication eligibility
   - Sends admission rejection email
   - Creates in-app notification

4. **Login Access**:
   - Only students with status "APPROVED" can login
   - PENDING students receive "ACCOUNT_PENDING_APPROVAL" error
   - REJECTED students receive "ACCOUNT_REJECTED" error
   - DEACTIVATED students receive "ACCOUNT_DEACTIVATED" error

## User Account Creation Point

**During registration**, not during approval:
- Located in `backend/src/controllers/student.controller.js`, `registerStudent` function
- Lines 242-249: Creates User document before Student document
- Password is hashed via User model pre-save hook
- Student document references User via `user_id` field

## Credential Generation Point

**Login credentials (password) are generated during registration**:
- Password provided in registration form is hashed and stored in User.collection
- **However, login access is granted only after approval**
- Enrollment number is generated during approval (`studentApproval.controller.js`, lines 163-177)

## Parent Account Creation Point

**After student approval**:
- Located in `backend/src/controllers/studentApproval.controller.js`, `approveStudent` function
- Lines 190-203: Calls `parentCreationService.createParentUsers(student)` 
- Service creates User accounts for parents with role "PARENT_GUARDIAN"
- Temporary passwords are generated and emailed to parents

|------------|------------|---------------------------------------------------|
| APPROVED   | ✅ Yes     | -                                                 |

## Security Risks

1. **Password exposure during registration**: Password is transmitted in plain text during registration (though hashed before storage)
2. **Temporary password for parent accounts**: ParentCreationService generates predictable temporary passwords ("Parent" + random string)
3. **No password change enforcement for students**: Students aren't forced to change password on first login (only applies to non-STUDENT roles)
4. **Orphaned User accounts**: If Student creation fails after User creation, the User is deleted (rollback mechanism exists)

## Business Risks

1. **Early credential availability**: Students receive login capability (via hashed password) immediately upon registration, but are blocked by status check. However, if status check fails, credentials exist.
2. **Payment not verified before approval**: System creates StudentFee record during approval but doesn't verify payment has been made before granting access.
3. **Enrollment number generation dependency**: Enrollment number depends on counts of existing approved students, which could have race conditions in high-volume scenarios.
4. **Parent account creation timing**: Parent accounts are created after student approval, but if approval fails after parent creation, orphaned parent accounts may exist.

## Root Cause

The system correctly separates:
- **Authentication** (User account with password) - created at registration
- **Authorization** (login access granted) - controlled by student status
- **Student lifecycle progression** - PENDING → APPROVED/REJECTED

However, there's a gap in verifying payment/enrollment completion before granting full access.

## Recommended Fix

1. **Add payment verification check**: Before approving student, verify that at least first installment payment has been received
2. **Add enrollment confirmation step**: Consider adding a separate "enrollment confirmed" status or verification step
3. **Strengthen parent account security**: Consider requiring password change on first login for parent accounts
4. **Add audit trail for credential generation**: Log when enrollment numbers are generated and to whom they're assigned

## Priority

**P0** - This is a core workflow investigation for student admission and enrollment. Understanding exactly where credentials are generated and when access is granted is fundamental to system security and proper functioning.
