# Archived Timetable Visibility — Architecture Review & Corrected Plan

## Executive Summary

The original implementation plan contains a **critical architectural flaw** in the proposed middleware chain for the archived endpoints. This review corrects that flaw and provides a prioritized implementation roadmap.

---

## Critical Finding: HOD Middleware Misapplication

**The Problem:**
The original plan proposes this middleware chain for `GET /api/timetable/archived`:
```
auth → role(ROLE.TEACHER, ROLE.HOD, ROLE.PRINCIPAL, ROLE.EXAM_COORDINATOR) → collegeMiddleware → hod
```

**Why this is wrong:**
The `hod` middleware (`backend/src/middlewares/hod.middleware.js`) is a HOD-only gatekeeper. At line 15-17, it finds the teacher profile. At line 80-92, if no timetableId is present, it resolves the department from `hod_id`. At line 64-75, if the teacher is NOT the HOD of the department, it throws:
- `"Only HOD can manage this timetable"` (403)
- `"HOD_WRONG_DEPARTMENT"` (403)

This means if a regular TEACHER hits the archived endpoint with `hod` middleware active, they receive 403 — even though the plan explicitly states regular teachers should see their courses' archived timetables. The middleware contradicts the stated requirement.

**Correct approach:**
The archived list endpoint should use the same middleware pattern as `getTimetables`:
```
auth → role(ROLE.TEACHER, ROLE.HOD, ROLE.PRINCIPAL, ROLE.EXAM_COORDINATOR) → collegeMiddleware
```
No `hod` middleware. Department/course scoping is handled inside the controller function, exactly as `getTimetables` does it (lines 309-337 of `timetable.controller.js`).

## 1. Archived Visibility Design

### 1.1 What Should Be Visible

| Status | Active Tab | Archived Tab | Notes |
|--------|-----------|-------------|-------|
| DRAFT | ✅ Yes | ❌ No | Editable in active view |
| PUBLISHED | ✅ Yes | ❌ No | Visible to students |
| ARCHIVED | ❌ No | ✅ Yes | Read-only, compliance record |

### 1.2 Read-Only Access Decision

**Decision: YES — archived timetables should be viewable in read-only mode.**

**Rationale:**
- ERP auditability requirement: HODs may need to review archived schedules for compliance, dispute resolution, or historical reference
- Attendance history is preserved in `AttendanceSession.slotSnapshot` — viewing the timetable without viewing the schedule provides incomplete audit trail
- The 404 guard on all read endpoints (`getTimetableById`, `getWeeklyTimetableById`, etc.) creates a visibility black hole
- Viewing is read-only — no mutation risk

**Implementation approach:**
- Modify `getTimetableById` and `getWeeklyTimetableById` to accept an optional query parameter or header that allows ARCHIVED access
- OR: create dedicated `GET /api/timetable/archived/:id` endpoint that permits ARCHIVED status
- Recommendation: modify existing endpoints with `?includeArchived=true` query param, keeping the default behavior (404) unchanged. This is minimally invasive.

## 2. HOD Access Model

### Current Behavior (getTimetables)

| Role | What they see | How |
|------|--------------|-----|
| HOD | All timetables in their department | `filter.department_id = teacher.department_id` |
| Regular TEACHER | Only timetables for courses they teach | `filter.course_id = { $in: teacherCourses }` |
| PRINCIPAL | All timetables in college | No additional filter |
| EXAM_COORDINATOR | All timetables in college | No additional filter |

### Archived Access Model (proposed)

| Role | What they see | How |
|------|--------------|-----|
| HOD | All ARCHIVED timetables in their department | `filter.department_id = teacher.department_id` + `status: "ARCHIVED"` |
| Regular TEACHER | Only ARCHIVED timetables for courses they teach | `filter.course_id = { $in: teacherCourses }` + `status: "ARCHIVED"` |
| PRINCIPAL | All ARCHIVED timetables in college | `status: "ARCHIVED"` only |
| EXAM_COORDINATOR | All ARCHIVED timetables in college | `status: "ARCHIVED"` only |

**Key principle:** The archived endpoint should mirror the scoping logic of `getTimetables` exactly, just with `status: "ARCHIVED"` instead of `status: { $ne: "ARCHIVED" }`.

### HOD Middleware Verdict

**DO NOT use `hod` middleware on archived list endpoint.**

The `hod` middleware is designed for mutation operations (create, update, delete, publish, archive) where only HODs should act. For read operations on archived records, the controller-level scoping (already proven in `getTimetables`) is sufficient and correct. Using `hod` middleware would incorrectly block regular teachers from viewing their own courses' archives.

The `hod` middleware IS appropriate for:
- `PUT /:id/archive` (archive action — already correctly applied)
- `PUT /:id/publish` (publish action — already correctly applied)
- `DELETE /:id` (delete action — already correctly applied)
- Any future `PUT /:id/restore` (restore action)

## 3. College Admin Access Model

**Current state:** `getTimetables` does NOT include COLLEGE_ADMIN in its role list. College admins cannot list timetables at all.

**Question:** Should COLLEGE_ADMIN see archived timetables?

**Recommendation:**
- If College Admin needs timetable visibility at all (currently they don't), add them to the role list with college-wide access (same as PRINCIPAL)
- If not needed, do not add them to the archived endpoint either — keep the access model consistent
- This is a product decision, not a technical one

## 4. Stats API Design

### Original Plan Assessment

The original plan proposes `GET /api/timetable/stats` returning:
```json
{ "total": 12, "published": 5, "draft": 4, "archived": 3 }
```

**Assessment:** Sound design. Four separate `countDocuments` queries are simple, correct, and performant with existing indexes.

### Improved Design

Add `lastUpdated` timestamp for frontend cache invalidation:
```json
{
  "total": 12,
  "published": 5,
  "draft": 4,
  "archived": 3,
  "lastUpdated": "2026-06-06T10:00:00Z"
}
```

**Middleware:** Same as archived list — `auth → role(...) → collegeMiddleware` (no `hod`). Controller applies same role-based scoping as `getTimetables`.

**Caching:** Stats are cheap to compute (4 indexed count queries). No caching needed at the API level. Frontend can cache in state for the session lifetime.

## 5. Frontend UX Design

### 5.1 Tab Structure

```
[Active Timetables] [Archived Timetables]
        ▲                    ▲
    default               hidden until
    selected              archive exists
```

**Tab behavior:**
- "Active Timetables" is the default tab, pre-selected on mount
- "Archived Timetables" tab is always visible (not hidden when count is 0 — hiding it would make it undiscoverable)
- Switching tabs triggers independent data fetch
- Each tab has its own loading, error, and empty states
- Stats bar is shared between tabs and sourced from the stats API

### 5.2 Stats Bar Placement

**Current:** Stats bar sits above the table inside the card.

**Keep:** Same placement. Stats are independent of tab selection — they always show aggregate counts.

### 5.3 Archived Table Actions

| Action | Allowed | Rationale |
|--------|---------|-----------|
| View (weekly schedule) | ✅ Yes | Compliance, audit trail |
| Edit | ❌ No | ARCHIVED is terminal state |
| Publish | ❌ No | Cannot re-publish archived |
| Delete | ❌ No | Cannot delete archived |
| Archive | ❌ No | Already archived |
| Restore | ❌ Not in this phase | Future enhancement |

## 6. Multi-Tenant Security

### Security Checklist

| Threat | Mitigation |
|--------|-----------|
| Cross-tenant data leak | All queries filter by `college_id: req.college_id` |
| Cross-department HOD access | Controller applies `department_id` filter for HOD role |
| Teacher sees other departments' archives | Controller applies `course_id` filter for regular teacher |
| Student accesses archived | Role middleware blocks STUDENT role |
| Unauthenticated access | `auth` middleware validates JWT + active status |
| Deleted college access | `collegeMiddleware` checks `college.isActive` |
| Direct API bypass | Backend guards are authoritative; frontend tabs are UX only |

### Index Verification

Existing indexes sufficient:
- `{ college_id: 1, status: 1 }` — supports stats counts and archived list queries
- `{ department_id: 1, status: 1, createdAt: -1 }` — supports HOD-scoped queries
- `{ course_id: 1, status: 1 }` — supports teacher course-scoped queries

No new indexes required.
