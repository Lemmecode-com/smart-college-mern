# Business Logic Review — Archived Timetable View Button Contradiction

## Problem Statement

The Archived Timetables tab displays a **View** button for each archived timetable. Clicking it navigates to `/timetable/:id/weekly`, which calls `getWeeklyTimetableById`. That controller explicitly returns HTTP 404 for any timetable with `status === "ARCHIVED"`. Result: every View click on an archived timetable produces a 404 error page.

---

## Option A: Remove/Disable View Button on Archived Tab

### Approach
Hide or disable the View button for all rows when `activeTab === "archived"`. The archived tab becomes a read-only list with no navigation — users can see that archived timetables exist (name, semester, academic year, status badge) but cannot access their details.

### Business Impact
- **Negative:** HODs lose ability to verify what was archived. After archiving a PUBLISHED timetable, the HOD has no way to confirm the archived data is correct without querying MongoDB directly.
- **Negative:** If a mistake was made during archiving (wrong timetable, wrong semester), there is no recovery path visible in the UI. The HOD must contact support or use direct DB access.
- **Neutral:** Active tab behavior is unaffected.

### Auditability Impact
- **Negative:** Archive becomes a black hole. The ERP audit trail records that a timetable was archived (AuditLog entry `TIMETABLE_ARCHIVED`), but the auditor cannot verify the archived content matches what was active. This creates an audit gap — the action is logged but the result is unverifiable from the UI.
- **Negative:** Compliance reviews (e.g., "show me what was scheduled in Fall 2025") require database-level queries instead of self-service UI access.

### ERP Compliance Impact
- **Negative:** Most ERP audit standards (SOX, ISO 27001, institutional accreditation) require that archived records remain accessible for a defined retention period. A 404 response on a legitimate archived record is a compliance risk — it suggests data loss when the data actually exists.
- **Negative:** Data retention policies assume archived data is retrievable. If retrieval always fails, the retention policy is theater.

### User Experience Impact
- **Negative:** User clicks View → sees 404 error page. This is a broken interaction. The user must use the back button to return to the archived list. Repeated 404s erode confidence in the system.
- **Neutral:** No loading state, no partial render — the error is immediate and unambiguous.
- **Positive (arguable):** The tab is simpler — fewer buttons, fewer decisions.

### Data Integrity Impact
- **Neutral:** No data is lost or corrupted. The data exists in MongoDB; it is simply not served.
- **Negative:** The UI creates an incorrect mental model that archived data is gone. Users may believe the archive workflow deletes data, undermining trust in the "preserved for record-keeping" promise made in the archive confirmation dialog.

### Verdict: NOT RECOMMENDED
This option trades a small amount of controller code removal for significant audit, compliance, and UX damage. The View button exists for a reason — removing it because the backend blocks it is treating the symptom, not the cause.

---

## Option B: Allow Read-Only Access to ARCHIVED Timetables

### Approach
Modify the five read endpoints that currently block ARCHIVED status (`getTimetableById`, `getWeeklyTimetableById`, `getSchedule`, `getTodaySchedule`, `getWeeklySchedule`) to permit access when `status === "ARCHIVED"`. The frontend Archived tab already hides all mutation buttons (Edit, Publish, Archive, Delete), so the access is effectively read-only by UX. Backend enforcement requires a separate mechanism — see "Enforcement Strategy" below.

### Enforcement Strategy

**Option B1 — Controller-level allowlist:**
```js
// In each of the 5 read controllers:
if (timetable.status === "ARCHIVED") {
  // Allow if request originates from archived context
  // (no reliable way to detect this from req alone)
}
```
Problem: The controller cannot distinguish "user is viewing from Archived tab" vs "user is on the active list and somehow got an ARCHIVED ID." The backend has no concept of "which tab the user clicked from."

**Option B2 — Separate endpoint for archived detail:**
Create `GET /api/timetable/archived/:id` (or reuse `GET /api/timetable/archived` with an ID param). This endpoint explicitly allows ARCHIVED status. The frontend Archived tab's View button calls this endpoint instead of the general `/:id/weekly` route.

Problem: Duplicates logic from `getWeeklyTimetableById`. Requires maintaining two parallel controller functions or extracting shared logic.

**Option B3 — Query parameter opt-in (RECOMMENDED):**
Modify the five read controllers to accept an optional query parameter:
```
GET /api/timetable/:id/weekly?includeArchived=true
```
When `includeArchived=true`, skip the ARCHIVED 404 guard. When absent or false, retain current behavior (404 for ARCHIVED).

Frontend change: Archived tab's View button navigates to `/timetable/${id}/weekly?includeArchived=true` or the API call includes `?includeArchived=true`.

Advantages:
- Minimal backend change — one conditional per controller
- No new routes or endpoint duplication
- Active tab behavior is unchanged (no `includeArchived` param → 404 as before)
- Clear intent: "this request explicitly opts into archived access"
- Future-proof: other features (audit reports, compliance views) can also use `includeArchived=true`

### Business Impact
- **Positive:** HODs can verify archived content immediately after archiving. "Archive and confirm" becomes a real workflow.
- **Positive:** Compliance reviewers can access archived timetables through the standard UI.
- **Positive:** No data migration, no schema change, no new collections.
- **Neutral:** Active tab behavior is unchanged.

### Auditability Impact
- **Positive:** The full audit trail becomes verifiable. Auditor can: (1) see AuditLog entry for archive action, (2) navigate to archived tab, (3) click View, (4) confirm the archived content matches the logged action.
- **Positive:** The `slotSnapshot` in AttendanceSession was designed for exactly this — preserving immutable historical data for audit. Making archived timetables viewable activates that design intent.
- **Neutral:** No new audit log entries needed for read-only access (read operations are typically not audited in this system).

### ERP Compliance Impact
- **Positive:** Archived records are retrievable, satisfying data retention and access requirements.
- **Positive:** The system's own archive confirmation message says "preserved for record-keeping." Option B makes that promise true.
- **Neutral:** No change to data retention policies or purge workflows.

### User Experience Impact
- **Positive:** View button works as expected. No 404 error page.
- **Positive:** Consistent UX — View button behaves the same way on both tabs. Users don't need to understand that "archived means invisible."
- **Neutral:** The archived tab's View navigates to the same WeeklyTimetable page. The page shows the schedule. Mutation buttons are hidden (already implemented). The user sees the data and can only read it.
- **Minor negative:** WeeklyTimetable.jsx may show edit/delete buttons that need to also respect archived status (see "Required Frontend Changes" below).

### Data Integrity Impact
- **Positive:** No data is lost. The data that already exists becomes accessible.
- **Positive:** Reinforces the immutability contract — ARCHIVED means "frozen and viewable," not "deleted."
- **Neutral:** No schema changes, no data migration.

### Required Frontend Changes (Option B3)

1. **TimetableList.jsx — View button URL:**
   Current: `navigate(\`/timetable/${t._id}/weekly\`)`
   Change for archived tab: `navigate(\`/timetable/${t._id}/weekly?includeArchived=true\`)`

2. **WeeklyTimetable.jsx — Mutation button guards:**
   The WeeklyTimetable page shows Edit/Delete/Publish/Archive buttons based on timetable status. Currently it checks `timetable.status === "PUBLISHED"` for edit disable, but does not check for ARCHIVED. If an ARCHIVED timetable is viewed, the page would show mutation buttons that would fail at the API level.
   
   Required: Add `timetable.status === "ARCHIVED"` to all mutation button disabled conditions in WeeklyTimetable.jsx.

## Recommendation

**Implement Option B with the query parameter approach (Option B3).**

### Rationale

1. **The system already promised visibility.** The archive confirmation dialog says: "It will be preserved for record-keeping and hidden from active schedules." "Preserved" implies accessible. The current 404 behavior violates that promise.

2. **The data is already there.** The AttendanceSession model has `slotSnapshot` — immutable historical data specifically designed for audit review. Blocking access to the timetable that owns those sessions makes the snapshots unreachable.

3. **The View button was intentionally added.** It exists in the archived tab's JSX (line 787). Its presence indicates a design intent to allow viewing. The 404 is a leftover from a phased implementation ("Phase 3" per the comment) that was completed before the archived tab was built.

4. **Query parameter is the least invasive change.** It requires no new routes, no new controllers, no schema changes. Five one-line conditionals and one frontend URL change. The default behavior (no parameter) remains unchanged for all existing callers.

5. **It is the ERP-standard pattern.** SAP, Oracle, and other enterprise systems distinguish between "active" and "archived" records at the business logic level — archived records are readable but not mutable. The `?includeArchived=true` pattern is common in ERP APIs.

### Implementation Scope

**Backend (5 controllers, 5 lines):**
- `getTimetableById` — line 244: add `&& !req.query.includeArchived` to the ARCHIVED guard
- `getWeeklyTimetableById` — line 882: same
- `getSchedule` — line 1086: same
- `getTodaySchedule` — line 1201: same
- `getWeeklySchedule` — line 1238: same

**Frontend (1 line change + 1 component guard):**
- `TimetableList.jsx` line 790: archived tab View button navigates to `/timetable/${t._id}/weekly?includeArchived=true`
- `WeeklyTimetable.jsx`: add `t.status === "ARCHIVED"` to all mutation button disabled conditions

### What Remains Unchanged
- Active tab behavior (no `includeArchived` param → 404 for ARCHIVED as before)
- All mutation endpoints (publish, archive, delete) — still block ARCHIVED
- All other read endpoints (student views, teacher schedules) — still filter out ARCHIVED
- The stats API, the archived list endpoint — unaffected

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Accidental mutation of archived data | Low | High | All 4 mutation endpoints already block ARCHIVED; WeeklyTimetable.jsx button guards added |
| Performance degradation from serving archived data | Very Low | Low | Same queries, same indexes, same data volume |
| Cross-department access to archives | Very Low | Low | `college_id` + `department_id` scoping already enforced before the ARCHIVED check |
| Breaking change to existing API consumers | None | None | Default behavior unchanged; `includeArchived` is opt-in |

---

## Alternative Considered and Rejected: Separate Archived Read Endpoint

Creating `GET /api/timetable/archived/:id` would work but:
- Duplicates query logic from `getWeeklyTimetableById` (department scoping, slot population, exception processing)
- Requires frontend to call a different URL for archived vs active — inconsistent UX
- Adds route maintenance burden (two endpoints that must stay in sync)
- The query parameter approach achieves the same result with less code duplication

---

## Decision

| Item | Decision |
|------|----------|
| Option chosen | **Option B — Read-only access via `?includeArchived=true`** |
| View button on archived tab | **Keep — make it work** |
| Backend changes | 5 controller conditionals |
| Frontend changes | 1 URL change + WeeklyTimetable mutation guards |
| Scope creep risk | Low — minimal, targeted changes |
| Alignment with ERP standards | High — matches SAP/Oracle archived-record pattern |
